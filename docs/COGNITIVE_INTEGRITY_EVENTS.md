# Cognitive Integrity Events

Chromie treats prompt or output truncation as a critical integrity failure, not a
transient inference error. It does not clear context, reduce context, retry the
same turn, or pass partial output to execution.

## Runtime coverage

The same event producer is used at all goal-driven cognitive boundaries:

- Goal Association;
- Fast Planner;
- Deep Planner; and
- Response Composer.

Every captured truncation declares that the result is untrusted, automatic retry
and context reduction are forbidden, and new execution is prevented.

## Chromie and data-loop boundary

Chromie classifies the event, gathers evidence, atomically writes a complete
package under `CHROMIE_EVENT_ROOT/ready/<event_id>`, and optionally writes a
lightweight trigger descriptor to `CHROMIE_DATA_LOOP_TRIGGER_ROOT`.

The external data-loop system owns deduplication, event and transport merging,
bandwidth and storage governance, retry, retention, and cloud delivery. The
ready event directory is the source of truth; the trigger is only notification.

## Event package

The initial package contains `event.json`, `failure.json`, `runtime.json`,
`model_exchange.json`, and `user_interaction.json`. It is first written under
`.staging` and then atomically renamed into `ready`.

The model exchange can contain sensitive user and runtime context. Product
privacy, redaction, encryption, retention, and access policies must govern it.

## Episodes and scenario mining

Incident capture and experience episodes are complementary:

- an incident is an immutable forensic snapshot at a precise failure boundary;
- an episode is a longitudinal conversation and behavior record;
- a scenario candidate is a derived regression proposal.

The existing `EpisodeRecorder` and `scripts/evaluate_experience_episodes.py`
remain offline derivation tools. The data loop may upload both incident packages
and episode records and correlate them through session, conversation, turn, and
time fields. Scenario mining should use the correlated evidence later; it must
not run synchronously in the safety-critical failure path.

A generated candidate may assert, for example, that truncation produces no new
execution, no automatic retry, a complete incident package, and a deterministic
user notification. Candidate scenarios require review and must never be promoted
automatically from one event.

## User notification

Failure wording is deterministic and is not generated by the failed model.
Chromie may state that evidence was recorded and handed to the diagnostic system
only after the trigger descriptor was accepted. It must not claim cloud upload,
because only the external data loop can establish that fact.
