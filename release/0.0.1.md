# Chromie 0.0.1 Candidate Release Notes

> **Status:** blocked candidate, not a published release.
>
> **Scope:** proposed Chromie `0.0.1` release. Chromie is the real host control plane;
> robot execution evidence for this release is limited to Soridormi MuJoCo
> `sim` mode. The candidate proposes generated-speech voice regression,
> structured text interaction, and the pinned Soridormi contract. It does not
> claim human physical microphone/speaker support or physical robot support.

## Supported Surface

- realtime host audio loop, VAD, ASR coordination, TTS generation, ordered
  playback, and deterministic interruption paths;
- native strict `InteractionResponse` output with explicit compatibility
  rollback;
- trusted host Skill Runtime scheduling, request-bound confirmation,
  cancellation, timeout, and trace evidence;
- generated-speech voice acceptance through `synthetic`, `virtual-mic`, and
  `acoustic` modes;
- text-to-MuJoCo and structured speech/text routing into Soridormi named
  skills using the pinned capability contract;
- TaskGraph validation and gated read, planning, guarded, and physical-policy
  execution paths, with physical execution gates remaining off.

## Explicit Limitations

- This is not a production physical-robot release.
- This is not a human voice-device support release. The acoustic runner can
  exercise a configured speaker/input path with generated speech, but human
  pronunciation, room noise, operator observation, and physical audio-device
  support still require a clean `supervised` evidence bundle.
- The host hardware daemon remains a mock compatibility path.
- Jetson profiles are configuration profiles, not a verified Jetson
  distribution.
- The common safe base enables structured interaction and authoritative
  goal-driven `apply` for `chat`; Soridormi remains off there. The maintained
  Soridormi launcher enables that provider and widens authority to
  `chat,robot_action`.
- The current acceptance runner records a user-supplied paired Soridormi
  checkout but cannot bind it to the source executing behind the MCP endpoint.
- Running Chromie service images and loaded models are not yet bound to the
  candidate source revision, and maintained release image inputs still use
  mutable `latest` tags.

## Required Publication Evidence

This remains a candidate. The compatibility manifest blocks publication until
new clean evidence is retained from the corrected goal-driven authority path at
the exact release revisions.

- `./scripts/run_tests.sh` passes from the tagged revision.
- `python scripts/check_docs.py` passes.
- The behavior scenario suite passes with `--no-write`.
- Automated voice evidence passes `scripts/verify_voice_evidence.py
  --allow-automated --require-clean` and reports
  `policy_evaluation_ready=true` from an endpoint-reported Soridormi revision.
- Text-to-MuJoCo or retained simulator evidence records exact Chromie and
  Soridormi manifest/paired-checkout revisions, a matching endpoint-reported
  Soridormi revision, ordered goal-driven Soridormi `sim` execution,
  completion, and safe idle.
- Release preparation accepts that simulator evidence through
  `--text-mujoco-summary` and writes
  `cognitive-runtime-acceptance.json` into the candidate bundle.
- Release artifacts declare exact Chromie and Soridormi revisions and contain
  no secrets, execution tokens, or private environment values.
- Running Chromie images/models are bound to the candidate revision, and all
  publishable image references are immutable.

## Upgrade and Rollback

For this first release, upgrade means checking out the exact `0.0.1` tag,
rebuilding the runtime environment, and rebuilding/restarting services. Roll
back by checking out the previous known revision and restoring generated
runtime configuration from source files.

An intentional rollback from the unified goal-driven path to the legacy Agent
adapter requires both host and Agent switches; it does not change `/run`
callers:

```env
ORCH_COGNITIVE_RUNTIME_MODE=off
ORCH_LEGACY_SEMANTIC_FALLBACK_ENABLED=0
AGENT_INTERACTION_OUTPUT_MODE=legacy-adapter
AGENT_NATIVE_INTERACTION_FALLBACK=0
AGENT_LEGACY_CAPABILITY_FALLBACK_ENABLED=0
```

Do not enable validation fallback as a permanent compatibility mode; use the
explicit adapter mode when rollback is intentional.

## Reproducible Build Provenance

A publishable release requires immutable container references, exact direct
dependency pins, and `release/model-lock.json` for immutable ASR/TTS snapshots.
The current mutable development aliases must be replaced first. A publishable
bundle must include complete `build-provenance.json` evidence with resolved
Docker image IDs or repository digests, transitive Python dependencies, and
installed Ollama model digests.
