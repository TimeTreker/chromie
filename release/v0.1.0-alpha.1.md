# Chromie v0.1.0-alpha.1 — Candidate Release Notes

> **State:** prepared candidate, not yet published. A real release requires a
> clean revision and a passing M13 microphone/MuJoCo evidence bundle.

## Scope

This alpha candidate targets one documented Linux x86_64 NVIDIA reference host
and provides:

- realtime microphone, VAD, ASR, routing, native structured Agent output, TTS,
  playback, and barge-in;
- native strict `InteractionResponse` output with a default-off legacy fallback;
- trusted host Skill Runtime scheduling, cancellation, confirmation checks, and
  traces;
- MuJoCo-backed Soridormi named skills using the pinned capability contract;
- TaskGraph validation and gated read, planning, guarded, and physical-policy
  execution paths;
- synthetic, virtual-microphone, supervised evidence, and release tooling for
  staged alpha validation.

## Explicit limitations

- This is not a production physical-robot release.
- The host hardware daemon remains a mock compatibility path.
- Jetson profiles are not a verified Jetson distribution.
- Spoken request-bound confirmation is implemented, but clean automatic and
  supervised approval/denial evidence is still required before publication.
- Microphone, speaker, GPU, and MuJoCo behavior must be validated on the exact
  reference host used for the release.
- Structured host rollout remains opt-in until the retained evidence is reviewed.

## Required publication evidence

- `./scripts/run_tests.sh` passes from the tagged revision.
- Automatic `synthetic` and `virtual-mic` matrices pass as regression evidence.
- `scripts/m13_voice_acceptance.py --mode supervised` records all seven required
  cases on the reference host.
- `scripts/verify_m13_evidence.py --require-clean` passes that supervised bundle.
- The release bundle declares exact Chromie and Soridormi revisions.
- Operator notes confirm safe idle/recovery after cancellation or stop cases.
- No secrets, execution tokens, or private environment values are present.

## Upgrade and rollback

For the first alpha, upgrade means checking out the exact tag, rebuilding the
runtime environment, and rebuilding/restarting the services. Roll back by
checking out the previous known revision and restoring its generated
`.env.runtime` from source configuration.

The structured Agent path can be rolled back without changing `/run` callers:

```env
AGENT_INTERACTION_OUTPUT_MODE=legacy-adapter
AGENT_NATIVE_INTERACTION_FALLBACK=0
```

Do not enable validation fallback as a permanent compatibility mode; use the
explicit adapter mode when rollback is intentional.
